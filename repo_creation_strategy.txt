# Repository Provisioning Automation Using GitHub Template Repository and GitHub Actions

## 1. Overview

In our development process, new GitHub repositories are created periodically for different services, tools, and infrastructure components. Each newly created repository must follow a consistent organizational standard. These standards include repository structure, access control, branch management, and security protections.

Currently, when a new repository is created, several manual steps must be performed:

* Creating the repository
* Adding required teams and collaborators
* Creating standard branches
* Configuring default branch
* Applying branch protection rules
* Ensuring repository settings follow organization guidelines
* Maintaining consistent folder structure and templates

Performing these steps manually introduces several challenges:

* Risk of configuration inconsistencies
* Human errors during setup
* Time-consuming repetitive tasks
* Lack of standard enforcement across repositories

To address these issues, we are implementing a **standardized repository provisioning solution** using:

1. **A GitHub Template Repository** (to standardize repository contents)
2. **A GitHub Actions provisioning workflow** (to automate repository creation and configuration)

This approach allows us to automate the creation of repositories while ensuring that all repositories follow the required standards.

---

# 2. Goals of the Solution

The primary goals of this solution are:

* Standardize the structure and configuration of all repositories
* Reduce manual effort when creating repositories
* Ensure consistent access control and branch governance
* Enforce repository standards across teams
* Improve traceability and auditability of repository creation
* Enable a scalable approach for creating repositories on demand

---

# 3. High-Level Architecture

The solution consists of two major components:

### 1. Template Repository

A **template repository** that contains the standard structure, files, and configuration required for new repositories.

### 2. Repository Provisioning Automation

A **central GitHub Actions workflow** that provisions new repositories using the template repository and applies required configurations.

The overall architecture looks like this:

```
User
  │
  │ (Manual Trigger)
  ▼
Provisioning Repository
(GitHub Actions Workflow)
  │
  │ Creates Repo From Template
  ▼
Template Repository
  │
  ▼
New Repository Created
  │
  ├── Branches Created
  ├── Teams Assigned
  ├── Users Assigned
  ├── Branch Protections Applied
  └── Repository Settings Configured
```

---

# 4. Solution Components

## 4.1 Template Repository

The **template repository** serves as the golden standard for all newly created repositories.

This repository contains all files, folder structures, and reusable components that should be present in every repository.

### Typical contents of the template repository

Examples include:

* README template
* `.gitignore`
* `CODEOWNERS`
* Contribution guidelines
* Pull request templates
* Issue templates
* Documentation structure
* Standard folder structure
* Starter CI workflow files
* Security documentation
* License files (if applicable)

The template repository ensures that every new repository begins with the same structure and best practices.

Important note:
The template repository **does not control repository permissions or branch protections**. Those settings are applied separately by the provisioning automation.

---

## 4.2 Repository Provisioning Repository

A dedicated repository will be created to host the automation workflow responsible for provisioning new repositories.

Example name:

```
repo-provisioning
```

This repository contains:

* GitHub Actions workflow
* Helper scripts used by the workflow
* Configuration templates
* Documentation describing the provisioning process

This separation ensures that the provisioning logic is maintained independently from the template repository.

---

# 5. Repository Provisioning Workflow

The provisioning workflow is implemented as a **manually triggered GitHub Actions workflow**.

The workflow uses the `workflow_dispatch` trigger so that users can provide inputs such as repository name and access configuration when creating a repository.

Example trigger:

```
workflow_dispatch
```

This allows authorized users to run the workflow directly from the GitHub Actions interface.

---

# 6. Workflow Inputs

When triggering the provisioning workflow, users provide required parameters for repository creation.

Typical inputs include:

| Parameter             | Description                        |
| --------------------- | ---------------------------------- |
| repository_name       | Name of the new repository         |
| description           | Description of the repository      |
| visibility            | private or public                  |
| template_repository   | Template repository to use         |
| default_branch        | Default branch name                |
| admin_teams           | Teams with admin access            |
| write_teams           | Teams with write access            |
| admin_users           | Individual users with admin access |
| collaborators         | Additional collaborators           |
| create_develop_branch | Whether to create develop branch   |
| create_release_branch | Whether to create release branch   |

These inputs allow flexible configuration while maintaining standardization.

---

# 7. End-to-End Provisioning Process

The provisioning workflow executes several steps in sequence.

### Step 1 — Input Validation

The workflow first validates all inputs provided by the user.

Validation checks may include:

* Repository name format
* Whether repository already exists
* Valid team names
* Valid user names
* Template repository availability

If validation fails, the workflow stops and reports the error.

---

### Step 2 — Repository Creation from Template

The workflow creates a new repository using the GitHub **template repository API**.

This step copies:

* files
* directory structure
* GitHub workflow files
* documentation templates

The new repository is created within the organization.

---

### Step 3 — Repository Settings Configuration

After the repository is created, standard repository settings are applied.

Examples include:

* repository description
* repository topics
* enabling/disabling issues
* enabling/disabling wiki
* merge strategy configuration
* branch deletion settings

These settings ensure consistency across repositories.

---

### Step 4 — Branch Creation

If required by the standard branching strategy, additional branches are created.

Example standard branches:

```
main
develop
release
```

These branches are created using the commit reference from the default branch.

---

### Step 5 — Default Branch Configuration

The workflow ensures the correct default branch is set for the repository.

Typically:

```
main
```

or

```
develop
```

depending on the team’s branching strategy.

---

### Step 6 — Team Access Assignment

Standard team permissions are applied automatically.

Example:

| Team        | Permission |
| ----------- | ---------- |
| repo-admins | admin      |
| dev-leads   | maintain   |
| developers  | write      |
| auditors    | read       |

Using teams instead of individual users simplifies access management.

---

### Step 7 — Individual Collaborator Assignment

If specific individuals require access to the repository, they are added as collaborators with appropriate permissions.

Example:

```
admin access
write access
read access
```

Individual access should only be used when team access is not sufficient.

---

### Step 8 — Branch Protection / Ruleset Configuration

Critical branches must be protected to ensure code quality and security.

Protection rules may include:

* Pull request required before merge
* Minimum number of approvals
* Required status checks
* Dismiss stale reviews
* Prevent force pushes
* Restrict branch deletion
* Enforce code owner reviews

Example protected branches:

```
main
develop
release/*
```

GitHub Rulesets are recommended where available because they provide centralized governance and more flexible policy management.

---

### Step 9 — Workflow Summary and Logging

At the end of execution, the workflow publishes a summary containing:

* repository name
* repository URL
* branches created
* teams assigned
* users assigned
* branch protections applied
* workflow status

This provides visibility and auditability of repository provisioning.

---

# 8. Permission and Access Model

Access to repositories should primarily be managed through **GitHub Teams** rather than individual user assignments.

Recommended access model:

| Role               | Access         |
| ------------------ | -------------- |
| Platform Admins    | Admin          |
| Development Leads  | Maintain/Admin |
| Developers         | Write          |
| Security Reviewers | Read           |

This approach improves scalability and simplifies future access management.

---

# 9. Authentication Strategy for Automation

The provisioning workflow requires elevated permissions to perform actions such as:

* creating repositories
* managing repository permissions
* creating branches
* configuring branch protection

Authentication can be implemented using one of the following approaches:

### Option 1 — GitHub App (Recommended)

Provides secure and scoped permissions for organization automation.

### Option 2 — Personal Access Token (PAT)

Simpler setup but broader permission scope.

For enterprise environments, **GitHub Apps are generally preferred** due to better security and permission management.

---

# 10. Error Handling and Failure Scenarios

Failure scenarios may occur during repository provisioning.

Examples include:

* repository already exists
* invalid team name
* permission assignment failure
* branch protection configuration errors

If a failure occurs:

* The workflow will stop execution.
* A detailed error message will be logged.
* The workflow summary will clearly indicate the failed step.

Repositories will not be automatically deleted if partial configuration fails, allowing administrators to inspect and resolve issues.

---

# 11. Logging and Auditability

All provisioning actions are logged through GitHub Actions workflow logs.

The following information is captured:

* workflow trigger user
* execution timestamp
* workflow inputs
* repository creation details
* access assignments
* applied protections

This ensures traceability for repository creation events.

---

# 12. Governance and Workflow Access Control

Since repository creation affects the entire organization, execution of the provisioning workflow should be restricted.

Recommended control mechanisms:

* Limit write access to the provisioning repository
* Allow only authorized administrators to trigger workflows
* Use GitHub environment approvals if necessary

These controls prevent unauthorized repository creation.

---

# 13. Limitations of the Solution

While the template + automation approach provides strong standardization, some limitations exist.

### Template repository limitations

Template repositories only replicate repository contents such as files and folders. They do not automatically replicate:

* repository permissions
* branch protections
* organization policies

These must be applied separately by automation.

### Existing repositories

Repositories created before this automation will not automatically adopt new standards.

To address this, a future enhancement may include a **repository compliance workflow** that periodically validates repository configuration.

# 🟦 **STORY 1**

## 🧾 **Title**

Create Central Repository for Standardized Dependency Versions (Single Source of Truth)

---

## 🎯 **Objective**

Establish a **central repository** that maintains a **standardized list of approved (vulnerability-free) dependency versions** to be used across all service repositories.

This repository will act as the **single source of truth** for dependency versions and will help ensure consistency and reduce repeated vulnerability fixes across multiple repos.

---

## 🧩 **Problem Statement**

Currently, we have **20+ service repositories**, each maintaining its own dependency versions. This leads to:

* Inconsistent versions across repositories
* Repeated vulnerability issues for the same libraries
* Manual effort to fix the same issue in multiple repos
* No centralized visibility of approved versions

---

## 🧠 **Solution Approach**

Create a central repository that:

* Stores standardized dependency versions
* Tracks approved (vulnerability-free) versions
* Can be referenced by other tools/scripts (future stories)

---

## 📦 **Repository Details**

**Suggested Name:**

```
platform-dependency-management
```

---

## 📁 **Repository Structure**

Example structure:

```
platform-dependency-management/
  ├── README.md
  ├── dependencies.yaml
```

---

## 📄 **Dependency File Format**

Define a structured file (YAML preferred):

```yaml
dependencies:
  spring-boot: 3.2.5
  log4j: 2.21.1
  jackson: 2.15.3
  lodash: 4.17.21
```

---

## 📋 **Detailed Tasks**

### 🔹 Step 1: Create Repository

* Create repo under organization
* Set appropriate visibility (private)
* Add initial README

---

### 🔹 Step 2: Define Dependency File

* Create `dependencies.yaml`
* Define structure for dependencies
* Add initial set of commonly used libraries

---

### 🔹 Step 3: Identify Common Dependencies

* Review 3–5 existing service repos
* Extract commonly used dependencies
* Add them to central file

---

### 🔹 Step 4: Documentation

Update README with:

* Purpose of repo
* How to update dependencies
* Guidelines for adding new libraries

---

## ✅ **Acceptance Criteria**

* [ ] Central repository is created
* [ ] `dependencies.yaml` file is added
* [ ] At least 5 common dependencies are listed
* [ ] Structure is clear and consistent
* [ ] README explains purpose and usage
* [ ] Repo is accessible to team

---

## 🏁 **Definition of Done**

* Repository is created and shared with team
* Dependency file is finalized and reviewed
* Initial dependencies are validated against existing repos
* Documentation is complete and understandable by a new team member

---

## ⚠️ **Important Notes**

* Only include **approved and verified versions**
* Avoid adding unused dependencies
* Keep structure simple and extensible

# 🟩 **STORY 2**

## 🧾 **Title**

Build Script to Scan Repositories and Report Dependency Version Mismatches

---

## 🎯 **Objective**

Develop a script that scans all service repositories, compares their dependency versions against the central repository, and generates a report highlighting mismatches.

---

## 🧩 **Problem Statement**

Even after defining standardized versions (Story 1), we currently have:

* No visibility into which repos are outdated
* No automated way to identify mismatches
* Manual effort required to check each repo

---

## 🧠 **Solution Approach**

Create a script that:

* Reads standardized versions from central repo
* Scans each service repository
* Compares dependency versions
* Generates a report showing:

  * Matching versions
  * Outdated versions
  * Missing dependencies

---

## 🔗 **Dependencies**

* Requires completion of **Story 1 (central repo + dependency file)**

---

## 📋 **Detailed Tasks**

### 🔹 Step 1: Identify Target Repositories

* Prepare list of all service repos (~20+)
* Store repo names in config file or script

---

### 🔹 Step 2: Script Setup

* Choose language (Python preferred)
* Set up basic project structure

---

### 🔹 Step 3: Read Central Dependency File

* Parse `dependencies.yaml`
* Store standardized versions in memory

---

### 🔹 Step 4: Repository Scanning Logic

For each repo:

* Clone or pull latest code
* Detect project type:

  * Maven → `pom.xml`
  * Node → `package.json`
* Extract dependency versions

---

### 🔹 Step 5: Comparison Logic

* Compare repo version vs central version
* Categorize:

  * Match
  * Outdated
  * Not present

---

### 🔹 Step 6: Report Generation

Generate output in readable format:

Example:

```
Repo: service-A
  spring-boot → Outdated (3.1.0 → 3.2.5)
  log4j → Match
  jackson → Missing
```

Optional formats:

* Console output
* CSV file
* JSON report

---

### 🔹 Step 7: Testing

* Run script on 2–3 repos
* Validate accuracy of report

---

### 🔹 Step 8: Documentation

* Add README for script usage
* Include:

  * How to run
  * Input requirements
  * Sample output

---

## ✅ **Acceptance Criteria**

* [ ] Script reads dependency file from central repo
* [ ] Script scans at least one repository successfully
* [ ] Dependency comparison works correctly
* [ ] Report clearly shows mismatches
* [ ] Tested on minimum 2 repositories
* [ ] Documentation is available

---

## 🏁 **Definition of Done**

* Script runs end-to-end without errors
* Accurate mismatch report is generated
* Works for at least one project type (Maven or Node)
* Code is readable and maintainable
* README explains execution steps clearly

---

## ⚠️ **Important Notes**

* Do NOT modify repositories in this story (read-only scan only)
* Handle different project types carefully
* Ensure script can be extended in future (for auto-fix)
* Avoid hardcoding repo-specific values

---

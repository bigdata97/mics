# 📄 **Rollback Strategy for ArgoCD Deployments into GKE**

## **Overview**

As part of this story, I analyzed and validated multiple rollback strategies for ArgoCD-based deployments in our GKE environment.

The goal was to identify a **standardized, feasible, and scalable rollback approach** that aligns with our **GitOps model** and can be implemented consistently across all repositories in the Member ID Card System (MICS) platform.

During this analysis, I tested rollback options directly using our ArgoCD applications such as:

* `mics-ui-components-app-dev`
* `mics-reference-data-service-app-dev`

---

## **Current Application Setup (Observed)**

From my testing:

* ArgoCD is configured with:

  * **Auto Sync enabled**
  * **Auto Prune enabled**

* Example:

  ```
  Sync Policy: Automated (Prune)
  Sync Status: OutOfSync from main (3e56ff6)
  Health Status: Healthy
  ```

* Repo structure:

  ```
  Repo: https://github.com/cvs-health-source-code/mics-ui-components.git
  Path: k8s-manifests
  Target Revision: main
  ```

This confirms we are following a **GitOps-driven deployment model**, where:
👉 Git = Source of Truth
👉 ArgoCD = Sync Engine

---

# 🔁 **Strategy 1: Git Revert-Based Rollback (Recommended Approach)**

## **What I Tested**

I verified that ArgoCD is continuously syncing from Git (`main` branch), so any change in Git automatically triggers deployment.

---

## **Implementation Steps**

### Step 1: Identify last stable commit

```bash
git log --oneline
```

---

### Step 2: Revert the bad commit

```bash
git revert <commit-id>
```

---

### Step 3: Push to correct branch

```bash
git push origin develop   # DEV
git push origin release   # QA/UAT
git push origin main      # PROD
```

---

### Step 4: ArgoCD auto-sync

From my observation:

```
Sync Policy: Automated (Prune)
```

👉 This means ArgoCD will automatically:

* Detect change
* Sync application
* Apply rollback

---

### Step 5: Verification

Commands I used:

```bash
kubectl get pods -n mics-nonprod-dev
kubectl describe deployment mics-ui-components-deploy-dev -n mics-nonprod-dev
```

---

## ✅ **Pros**

* Fully aligns with **GitOps**
* No manual intervention in cluster
* Works across all environments
* Easy to standardize across repos
* Full audit trail in Git

---

## ❌ **Cons**

* Slight delay compared to direct rollback
* Requires Git access / PR approvals

---

## ✅ **Final Decision**

👉 This is selected as the **PRIMARY rollback strategy**

---

# ⚡ **Strategy 2: ArgoCD Revision-Based Rollback**

## **What I Tested**

I ran:

```bash
argocd app history mics-ui-components-app-dev
```

Observed output:

```
ID   DATE                      REVISION
3    2026-04-21               development-branch (3ca131a)
4    main (106319c)
5    main (106319c)
6    main (63e35a8)
```

---

## **Rollback Command**

```bash
argocd app rollback mics-ui-components-app-dev <revision-id>
```

---

## ✅ **Pros**

* Very fast rollback
* No Git changes required
* Useful for immediate recovery

---

## ❌ **Cons (Major Observations)**

From practical usage:

* Every user must:

  * Install ArgoCD CLI
  * Configure access
  * Login every time:

    ```bash
    argocd login internal-argocd-non-prod.mics-nonprod-use4.cvshealth.com --insecure --grpc-web
    ```

* Not scalable across team

* Breaks GitOps:

  * Git still contains bad version

* Requires manual reconciliation later

---

## ⚠️ **Important Observation**

If Git is not updated:
👉 ArgoCD will re-deploy bad version again

---

## ✅ **Conclusion**

👉 Technically works
👉 But **not feasible for org-wide usage**

# ⚙️ **Strategy 3: Kubernetes Rollout Undo**

## **What I Tested**

```bash
kubectl rollout history deployment mics-ui-components-deploy-dev -n mics-nonprod-dev
```

Rollback:

```bash
kubectl rollout undo deployment mics-ui-components-deploy-dev -n mics-nonprod-dev
```

---

## 🚨 **Critical Observation**

Since ArgoCD is enabled with auto-sync:

👉 Immediately after rollback:

* ArgoCD detects drift
* Re-applies latest Git state

---

## 🔧 **Workaround Tested**

### Step 1: Suspend ArgoCD

```bash
argocd app suspend mics-ui-components-app-dev
```

---

### Step 2: Perform rollback

```bash
kubectl rollout undo deployment mics-ui-components-deploy-dev -n mics-nonprod-dev
```

---

### Step 3: Resume ArgoCD

```bash
argocd app resume mics-ui-components-app-dev
```

---

## ✅ **Pros**

* Immediate rollback
* Works even without Git

---

## ❌ **Cons**

* Breaks GitOps completely
* Not sustainable
* Risk of configuration drift
* Requires manual intervention

---

## ✅ **Conclusion**

👉 Not suitable as standard approach
👉 Can be used **ONLY for emergency production fixes**

---

# 🏁 **Final Recommendation**

| Strategy            | Decision   | Usage                  |
| ------------------- | ---------- | ---------------------- |
| Git Revert          | ✅ Selected | Standard rollback      |
| ArgoCD Rollback     | ⚠️ Limited | Emergency (temporary)  |
| Kubernetes Rollback | ⚠️ Rare    | Critical incident only |

---

## **Final Approach to be Implemented**

### ✅ Standard Flow

1. Revert commit in Git
2. Push to correct branch
3. ArgoCD auto-sync applies rollback

---

### ⚡ Emergency Flow

1. Use ArgoCD rollback OR Kubernetes rollback
2. Immediately fix Git
3. Sync again

---

## **Key Takeaway**

Since our system is fully GitOps-driven:

👉 **All final state must come from Git**
👉 Any direct cluster changes are temporary and must be reconciled

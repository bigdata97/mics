To check what IAM roles a Service Account has in GCP using `gcloud`, you generally check the IAM policy bindings of the project (or resource) and filter for that Service Account.

## 1. Check roles of a Service Account at Project Level

```bash
gcloud projects get-iam-policy PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:serviceAccount:SERVICE_ACCOUNT_EMAIL" \
  --format="table(bindings.role)"
```

### Example

```bash
gcloud projects get-iam-policy my-gcp-project \
  --flatten="bindings[].members" \
  --filter="bindings.members:serviceAccount:my-sa@my-gcp-project.iam.gserviceaccount.com" \
  --format="table(bindings.role)"
```

This will show output like:

```text
ROLE
roles/storage.admin
roles/pubsub.publisher
roles/bigquery.dataEditor
```

---

## 2. Get Full Detailed Output (JSON)

```bash
gcloud projects get-iam-policy PROJECT_ID \
  --format=json
```

Then search for the Service Account email.

Useful when:

* debugging inherited roles
* checking conditions
* auditing all permissions

---

## 3. Check Roles Across Organization / Folder / Project

### Organization Level

```bash
gcloud organizations get-iam-policy ORGANIZATION_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:serviceAccount:SERVICE_ACCOUNT_EMAIL" \
  --format="table(bindings.role)"
```

### Folder Level

```bash
gcloud resource-manager folders get-iam-policy FOLDER_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:serviceAccount:SERVICE_ACCOUNT_EMAIL" \
  --format="table(bindings.role)"
```

---

## 4. Check Which Members Have Access TO the Service Account

This is different.

If you want to know:

> who can impersonate or use this service account

Run:

```bash
gcloud iam service-accounts get-iam-policy SERVICE_ACCOUNT_EMAIL
```

Example:

```bash
gcloud iam service-accounts get-iam-policy \
my-sa@my-project.iam.gserviceaccount.com
```

This shows roles like:

* `roles/iam.serviceAccountUser`
* `roles/iam.serviceAccountTokenCreator`

assigned on the Service Account itself.

---

## 5. Quick One-Liner (Very Common)

```bash
PROJECT_ID="my-project"
SA="my-sa@my-project.iam.gserviceaccount.com"

gcloud projects get-iam-policy $PROJECT_ID \
--flatten="bindings[].members" \
--filter="bindings.members:$SA" \
--format="table(bindings.role)"
```

---

## 6. If You Want ALL Permissions (not just roles)

You can use Policy Troubleshooter or IAM Recommender, but roles are usually sufficient for DevOps/debugging tasks in GKE, Terraform, GitHub Actions WIF, Pub/Sub, Artifact Registry, etc.

For example in your MICS/GitOps setup, this is commonly used to verify whether a CI/CD Service Account has:

* `roles/container.developer`
* `roles/artifactregistry.writer`
* `roles/iam.workloadIdentityUser`
* `roles/storage.admin`
* `roles/pubsub.publisher`
* `roles/secretmanager.secretAccessor`

etc.

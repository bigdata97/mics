# Branch-Based Environment Deployment Strategy

**Project:** ID Cards Excellence
**Story:** TLIDCE-2020
**Type:** Spike – Research & Design
**Author:** DevOps Team

---

# 1. Objective

The objective of this spike is to design a **branch-based deployment strategy** where application deployments are automatically triggered to the correct environment based on the Git branch used.

This enables:

• automated CI/CD pipelines
• consistent environment promotion
• clear separation of DEV / QA / PERF / PROD deployments
• predictable release workflow

The outcome of this spike is a **complete implementation design** that another engineer can implement.

---

# 2. Problem Statement

Currently deployments are not explicitly tied to Git branch activity. This can lead to:

* unclear promotion paths between environments
* inconsistent deployment behavior
* manual intervention during releases

To resolve this, we propose a **branch-to-environment mapping strategy** integrated with CI pipelines and GitOps deployment tools.

---

# 3. Branch to Environment Mapping

| Git Branch | Environment | Purpose                                |
| ---------- | ----------- | -------------------------------------- |
| develop    | DEV         | Developer integration testing          |
| release/*  | QA / PERF   | System testing and performance testing |
| main       | PROD        | Production deployment                  |

Important note:

This strategy **does NOT automatically move code between branches**.

Promotion between environments is done using **Pull Requests**.

Example:

```
feature → develop → release/v1.0 → main
```

---

# 4. High-Level Architecture (CLEAR IMPLEMENTATION VIEW)

This architecture shows **how code moves from Git to Kubernetes environments**.

### Key Components

1. **Git Repository**

   * Developers push code to branches.

2. **CI Pipeline**

   * Triggered automatically based on branch.
   * Builds Docker image.
   * Tags image based on environment.
   * Pushes image to container registry.

3. **Container Registry**

   * Stores built Docker images.
   * Images are tagged by environment.

4. **Deployment System (ArgoCD / CD Pipeline)**

   * Pulls the new image version.
   * Updates Kubernetes deployment.

5. **Kubernetes Cluster (GKE)**

   * Runs application workloads.

# 6. CI Pipeline Design

CI pipeline must trigger based on branch conditions.

Example logic:

| Branch    | Action                       |
| --------- | ---------------------------- |
| develop   | Build image + deploy DEV     |
| release/* | Build image + deploy QA/PERF |
| main      | Build image + deploy PROD    |

Pseudo logic:

```
if branch == develop
   deploy DEV

if branch startsWith release/
   deploy QA/PERF

if branch == main
   deploy PROD
```

---

# 7. Example CI Pipeline YAML

Example structure (generic CI):

```yaml
name: Build and Deploy

on:
  push:
    branches:
      - develop
      - main
      - release/*

jobs:

  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v3

      - name: Build Docker Image
        run: |
          docker build -t my-app:${{ github.sha }} .

      - name: Push Docker Image
        run: |
          docker push my-app:${{ github.sha }}

  deploy-dev:
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest

    steps:
      - name: Deploy to DEV
        run: echo "Deploying to DEV environment"

  deploy-qa:
    if: startsWith(github.ref, 'refs/heads/release/')
    runs-on: ubuntu-latest

    steps:
      - name: Deploy to QA
        run: echo "Deploying to QA environment"

  deploy-prod:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest

    steps:
      - name: Deploy to PROD
        run: echo "Deploying to PROD environment"
```
